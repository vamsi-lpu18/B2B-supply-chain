1. Global Technical Conventions

1.1 Clean Architecture Per Service

Every microservice follows a 4-layer architecture:

API Layer
Controllers & Auth Policies
HTTP endpoints, request routing, authorization attributes
Application Layer
Use Cases & DTOs
Service interfaces, validators, MediatR handlers, data transfer objects
Domain Layer
Entities & Business Rules
Aggregate roots, enums, value objects, domain invariants
Infrastructure Layer
Persistence & Integrations
EF Core DbContext, repositories, HTTP clients, background workers

1.2 Outbox Message Contract

Field Type Description
MessageId GUID Unique identifier for the message
EventType string Event type name (e.g., OrderPlaced)
Payload JSON text Serialized event payload
Status enum Pending → Published → Failed
CreatedAtUtc DateTime When the event was created
PublishedAtUtc DateTime? When successfully published to RabbitMQ
RetryCount int Number of publish attempts
Error string? Last error message if failed

1.3 Redis Key Namespaces

Namespace Purpose TTL
idempotency:{key} Prevent duplicate command processing 24 hours
revoked:{jti} Token revocation check Token lifetime
cache:{service}:{key} Response caching Configurable

3. Gateway Service — LLD

3.1 Responsibilities

  • Upstream entry point for all client requests (port 5000)
  • Route mapping to 6 downstream microservices
  • JWT authentication pass-through for protected routes
  • Per-route and global rate limiting (keyed by Oc-Client header)
  • Request correlation via X-Correlation-Id injection
  • Round-robin load balancing across service instances

3.2 Route Mapping

Upstream Prefix Downstream Service Auth Rate Limit
/identity/* IdentityAuth :8001 Public (auth routes) / JWT Yes
/catalog/* CatalogInventory :8002 Optional / JWT Yes
/orders/* Order :8003 JWT Required Yes
/logistics/* LogisticsTracking :8004 JWT Required Yes
/payments/* PaymentInvoice :8005 Public (credit check) / JWT Yes
/notifications/* Notification :8006 Public (ingest) / JWT Yes

11. Order Lifecycle — Sequence Diagram

sequenceDiagram
    actor Dealer
    participant FE as Angular SPA
    participant GW as API Gateway
    participant ORD as Order Service
    participant PAY as Payment Service
    participant CAT as Catalog Service
    participant RMQ as RabbitMQ
    participant NOT as Notification

    Dealer->>FE: Place Order
    FE->>GW: POST /orders
    GW->>ORD: Forward request
    ORD->>PAY: GET /credit-check
    PAY-->>ORD: Credit result

    alt Credit Approved
        ORD->>CAT: POST /soft-lock (reserve stock)
        CAT-->>ORD: Lock confirmed
        ORD->>ORD: Status = Processing
        ORD->>RMQ: Publish order.OrderPlaced
        RMQ->>NOT: Consume event
        NOT->>NOT: Create notification
    else Credit Insufficient
        ORD->>ORD: Status = OnHold
        ORD->>RMQ: Publish order.CreditHoldRequired
        RMQ->>NOT: Notify Admin
    end

    ORD-->>GW: Order response
    GW-->>FE: Order created
    FE-->>Dealer: Show confirmation

12. Order Status — State Machine

stateDiagram-v2
    [*] --> Placed
    Placed --> Processing : Credit approved
    Placed --> OnHold : Credit insufficient
    Placed --> Cancelled : Dealer/Admin cancel

    OnHold --> Processing : Admin approves hold
    OnHold --> Cancelled : Admin rejects hold

    Processing --> ReadyForDispatch : Fulfillment complete
    Processing --> Cancelled : Cancel

    ReadyForDispatch --> InTransit : Dispatched
    ReadyForDispatch --> Cancelled : Cancel

    InTransit --> Delivered : Delivery confirmed
    InTransit --> Exception : Delivery issue

    Exception --> InTransit : Issue resolved
    Exception --> Cancelled : Unresolvable

    Delivered --> Closed : Auto-close
    Delivered --> ReturnRequested : Dealer requests return

    ReturnRequested --> ReturnApproved : Admin approves
    ReturnRequested --> ReturnRejected : Admin rejects

    ReturnApproved --> Closed
    ReturnRejected --> Closed

    Cancelled --> [*]
    Closed --> [*]

17. Error Handling Strategy

Error Type HTTP Status Handling
Validation failure 400 Bad Request FluentValidation → structured property-level error response
Domain rule violation 400 Bad Request InvalidOperationException caught by middleware
Unauthenticated 401 Unauthorized Missing/invalid JWT token
Insufficient role 403 Forbidden ASP.NET Core [Authorize(Roles)] policy
Resource not found 404 Not Found Repository returns null → controller returns NotFound
Duplicate (idempotency) 200 OK (cached) Redis idempotency key hit → return previous response
Outbox publish failure N/A (async) Retry with max attempts → mark Failed
Event consumer failure N/A (async) Redelivery → dead-letter queue after max retries
