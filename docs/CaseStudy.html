1. Executive Summary

The Supply Chain Management Platform is an enterprise-grade B2B SaaS application that digitizes the full dealer-to-delivery lifecycle for a manufacturing/distribution business. It handles dealer onboarding, product catalog management, order processing with credit-hold workflows, logistics tracking with agent assignment, GST-compliant invoicing, and multi-channel notifications — all built on a production-ready microservices architecture.

2. Business Problem & Context

2.1 The Problem

A B2B manufacturing company manages hundreds of dealers who place bulk orders for products. The existing process is manual — orders via phone/email, credit checks on spreadsheets, shipment tracking through WhatsApp, and invoices generated in Tally. This leads to:

  • Order errors — manual data entry causes wrong quantities, pricing mismatches, and duplicate orders
  • Credit risk — no real-time visibility into dealer outstanding balances, leading to credit overexposure
  • Delivery blind spots — no centralized tracking; dealers call warehouse staff for status updates
  • Invoice delays — GST-compliant invoices generated days after delivery, causing payment collection lags
  • No analytics — management lacks dashboards for revenue trends, top dealers, or inventory health

2.2 The Solution

A fully digital platform where:

  • Dealers self-register, browse the catalog, place orders, and track shipments through a web portal
  • Admins manage dealer approvals, product catalog, credit limits, and monitor analytics dashboards
  • Warehouse staff process orders and manage inventory with automated stock alerts
  • Logistics managers create shipments, assign delivery agents, and track last-mile delivery
  • Agents update shipment status in real-time during delivery

6. Design Patterns & Architectural Decisions

6.1 Saga Pattern — Order Orchestration

The Order service implements a Saga Orchestration pattern to coordinate the multi-step order creation workflow across Catalog (stock) and Payment (credit) services.

stateDiagram-v2
    [*] --> Started : Dealer places order
    Started --> CreditCheckInProgress : Stock reserved, checking credit
    CreditCheckInProgress --> CompletedApproved : Credit OK
    CreditCheckInProgress --> AwaitingManualApproval : Credit exceeded
    AwaitingManualApproval --> CompletedApproved : Admin approves
    AwaitingManualApproval --> CompletedRejected : Admin rejects
    AwaitingManualApproval --> CompletedCancelled : Dealer cancels
    CompletedApproved --> [*]
    CompletedRejected --> [*]
    CompletedCancelled --> [*]
Figure 2 — Order Saga State Machine
Why Saga instead of 2PC? With 6 independent databases, a Two-Phase Commit is not feasible — it would require all databases to support XA transactions and would create blocking locks across services. The Saga pattern provides eventual consistency through compensating transactions (e.g., releasing stock locks on cancellation, restocking on return approval).

6.2 Transactional Outbox Pattern

Each service writes domain events to an OutboxMessages table in the same database transaction as the business data. A background OutboxDispatcher polls this table and publishes events to RabbitMQ.

sequenceDiagram
    participant Svc as Service Logic
    participant DB as Service Database
    participant Disp as OutboxDispatcher
    participant RMQ as RabbitMQ

    Svc->>DB: BEGIN TRANSACTION
    Svc->>DB: INSERT Order + UPDATE Stock
    Svc->>DB: INSERT OutboxMessages (OrderPlaced)
    Svc->>DB: COMMIT

    Note over Disp: Polls every 5 seconds
    Disp->>DB: SELECT WHERE Status = Pending (batch 20)
    Disp->>RMQ: BasicPublish (routing key: order.orderplaced)
    Disp->>DB: UPDATE Status = Published
Figure 3 — Transactional Outbox Pattern — Guaranteed Event Delivery
Guarantee: The outbox pattern ensures that if the business transaction succeeds, the event will eventually be published — even if RabbitMQ is temporarily down. The dispatcher retries up to 5 times before moving a message to Failed status.

12. Performance, Scalability & Resilience

⏱️ Performance

Target p95 read latency < 300ms, write latency < 500ms. Redis caching for frequently read data. EF Core compiled queries for hot paths.

📈 Horizontal Scalability

Stateless API services scale via RoundRobin LB. RabbitMQ decouples producers from consumers. Each service independently deployable.

🛡️ Resilience

Circuit breaker (Polly) prevents cascade failures. Outbox pattern survives broker outages. Retry with exponential backoff on transient failures.

🔄 Eventual Consistency

Strong consistency within each service (SQL transactions). Eventual consistency across services via Outbox + RabbitMQ events.

🔒 Idempotency

Redis-backed idempotency keys prevent duplicate order processing from retries or double-clicks. Outbox dispatcher uses status tracking.

📊 Observability

X-Correlation-Id traces requests across all 6 services + gateway. Serilog structured logging. Health check endpoints per service.

14. Key Challenges & Solutions

ChallengeProblemSolution Implemented
Distributed Transactions Order placement touches 3 services (Catalog, Payment, Order) — can't use a single DB transaction Saga Orchestration pattern with OrderSagaCoordinator. Compensating transactions (stock release, credit reversal) on failure. Persisted saga state for auditability.
Event Delivery Reliability If RabbitMQ is down when an order is placed, the notification is lost Transactional Outbox pattern — events are persisted atomically with business data. Background dispatcher retries up to 5 times with dead-letter fallback.
Concurrent Stock Updates Two dealers ordering the last 10 units simultaneously could oversell Two-phase stock model: SoftLock (reserve) → HardDeduct (confirm on dispatch). Optimistic concurrency via EF Core row versioning.
Token Refresh Race Condition Multiple parallel API calls hitting 401 simultaneously, each trying to refresh the token Frontend errorInterceptor uses a BehaviorSubject gate: first 401 triggers refresh, all subsequent 401s queue and wait for the new token, then retry.
Cross-Service Email Resolution Notification service receives events with userId but needs email address — owned by Identity service Internal HTTP call from Notification → Identity API (protected by X-Internal-Api-Key header, not JWT).
Role-Adaptive Frontend 5 different roles need different dashboards, menus, and action buttons from the same codebase Signal-based AuthStore.role() drives conditional rendering throughout. Route guards prevent unauthorized access. API services return role-scoped data.
CORS in Development Angular (:4200) and Gateway (:5000) on different ports — browser blocks cross-origin requests Angular dev proxy (proxy.conf.json) forwards all API paths to :5000 as same-origin. CORS configured only on Gateway for production.

15. Lessons Learned & Interview Talking Points

15.1 Architecture & System Design

Why Microservices over Monolith?

Each domain (Identity, Catalog, Order, Logistics, Payment, Notification) has distinct scaling needs and deployment cadences. Orders need high availability; Notifications can tolerate queuing. Independent deployment reduces blast radius.

Why Saga over 2PC?

6 separate databases make 2PC impossible — it would require XA-compatible transactions across all DBs, blocking locks, and single-point-of-failure coordinator. Saga provides eventual consistency with compensating actions, maintaining high availability.

Why Outbox over Direct Publish?

Direct publish creates dual-write problem: DB write succeeds but RabbitMQ publish fails = lost event. Outbox guarantees atomic event persistence with business data. The dispatcher is the reliability bridge.

Why API Gateway?

Single entry point simplifies frontend (one base URL), centralizes cross-cutting concerns (auth, rate limiting, CORS, circuit breaker), and enables load balancing without client changes.

15.2 Frontend & Angular Specifics

Why Signals over NgRx?

Angular Signals provide reactive state management without the boilerplate of actions, reducers, effects, and selectors. For this app's state complexity (auth + cart + loading), signals with computed() are sufficient and more maintainable.

Why Functional Interceptors?

Angular 21 deprecates class-based interceptors. Functional HttpInterceptorFn interceptors are simpler, tree-shakable, and align with standalone component architecture — no NgModule registration needed.

How does Silent Token Refresh work?

Error interceptor catches 401 → calls /identity/api/auth/refresh with withCredentials: true (sends httpOnly cookie) → receives new JWT → updates AuthStore → retries original request. A BehaviorSubject ensures only one refresh happens if multiple requests fail simultaneously.

How is the CartStore persisted?

Uses Angular effect() to automatically serialize cart items to localStorage on every signal change. On initialization, it hydrates from localStorage with full sanitization and validation of each item.

15.3 Backend & .NET Specifics

How does the Order State Machine enforce valid transitions?

The OrderAggregate (DDD aggregate root) maintains a CanTransitionTo() method with an explicit transition map. Invalid transitions throw InvalidOperationException. Status history is tracked in a separate table for audit trail.

How is stock consistency maintained?

Two-phase model: SoftLock on order placement (reserves stock), HardDeduct on dispatch (permanently removes). If soft-lock expires before dispatch, the system re-locks and retries once. On cancellation, locks are released. On return approval, stock is restocked.

How are concurrent modifications handled?

EF Core optimistic concurrency with row version columns. On DbUpdateConcurrencyException, the operation returns false rather than throwing — the caller can retry or report conflict.

Summary: This project demonstrates end-to-end design and implementation of a production-grade distributed system — from microservice decomposition and event-driven integration to a premium Angular SPA with role-based access — solving real B2B supply chain problems with industry-standard patterns and practices.
