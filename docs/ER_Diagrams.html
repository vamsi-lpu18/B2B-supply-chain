1. Data Architecture Overview

Design Principle: Each microservice owns a dedicated SQL Server database. No direct cross-database joins or shared schemas. Inter-service data access is achieved exclusively through published REST APIs or asynchronous domain events via RabbitMQ.

9. Shared Outbox Pattern Schema

Every microservice includes an identical OutboxMessages table for guaranteed event delivery:

erDiagram
    OUTBOX_MESSAGES {
        guid MessageId PK
        string EventType "max-200"
        string Payload "JSON text required"
        string Status "Pending | Published | Failed"
        datetime CreatedAtUtc
        datetime PublishedAtUtc "nullable"
        int RetryCount
        string Error "max-2000 nullable"
    }
Figure 9 — Outbox Message Schema (Present in All 6 Databases)
FieldTypeMax LengthPurpose
MessageIdGUID (PK)—Unique identifier for idempotent publish
EventTypestring200Fully qualified event name (e.g., OrderPlaced)
Payloadnvarchar(max)UnlimitedJSON-serialized event payload
Statusenum → string32Pending → Published by dispatcher, or Failed after max retries
CreatedAtUtcdatetime—Written in same transaction as domain data
PublishedAtUtcdatetime?—When successfully published to RabbitMQ
RetryCountint—Number of publish attempts by background worker
Errorstring?2000Last error message for failed publishes

11.3 Enum Storage Convention

Convention: All enum fields are stored as string (not int) using EF Core HasConversion<string>(). This improves query readability in SQL Server Management Studio and avoids breaking changes when enum values are reordered.
