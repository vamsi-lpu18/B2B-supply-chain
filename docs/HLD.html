1. Executive Summary & Business Context

The Supply Chain Management Platform is an enterprise-grade B2B SaaS solution that digitizes the complete dealer lifecycle — from onboarding and catalog browsing to order placement, logistics tracking, invoicing, and payment collection.

The system is built on a microservices architecture with 6 independently deployable domain services, an API gateway, and a modern Angular SPA. It leverages event-driven communication via RabbitMQ for cross-domain consistency, Redis for caching and idempotency, and SQL Server for transactional persistence.

Key Business Capabilities: Dealer onboarding & credit management · Product catalog with inventory control · Order lifecycle with credit-hold decisioning · Shipment tracking with agent assignment · GST-compliant invoice generation · Multi-channel notification delivery

2. Scope Definition

2.1 In Scope

  • Identity & access management with JWT authentication and RBAC
  • Product catalog with category hierarchy and stock tracking
  • Order lifecycle management with state machine transitions
  • Logistics shipment orchestration with agent/vehicle assignment
  • Payment gateway integration (Razorpay), credit management, invoice PDF generation
  • Multi-channel notification engine (in-app + email)
  • API gateway with rate limiting, correlation tracking, and load balancing
  • Role-based Angular SPA with real-time dashboards

2.2 Out of Scope

  • External ERP/SAP integration beyond current API contracts
  • Multi-region active-active deployment
  • Mobile native applications (iOS/Android)
  • Advanced ML-based fraud detection models

4. Architecture Principles

PrincipleDescription
Domain-Driven DecompositionEach microservice owns a single bounded context with its own database
Database-per-ServiceNo shared schemas — services communicate via APIs and events only
Outbox PatternReliable event publication using transactional outbox with background dispatchers
Event-Driven IntegrationRabbitMQ topic exchange for asynchronous cross-domain communication
CQRS-LiteMediatR pipeline with validation, idempotency, transaction, and logging behaviors
Zero-Trust AuthJWT validation at gateway and every service, with internal API key protection
Idempotent OperationsRedis-backed idempotency keys prevent duplicate processing

15.4 Guardrails & Governance

  • AI output is validated against strict schema before any action is visible or executable
  • Policy rules are applied after AI suggestion (role permissions, ownership checks, domain constraints)
  • Human approval required for medium and high impact changes
  • Auto-fallback to deterministic rule-based playbook if AI fails, times out, or confidence < threshold
  • Full audit trail: prompt context hash, model version, recommendation, approver, execution result
  • AI provider key stored in environment variable — never in source control

16. Risks & Mitigations

RiskImpactLikelihoodMitigation
RabbitMQ downtime causes event lossHighLowOutbox pattern ensures events are persisted before publish; dispatcher retries
Internal API keys exposedHighLowEnvironment variable storage; network-level access control planned
Single-region SQL Server failureCriticalLowRegular backups; multi-region considered for future phases
Token theft enables unauthorized accessHighMediumShort-lived tokens, Redis revocation store, refresh token rotation
Notification email delivery failureMediumMediumFailure-state tracking; retry with dead-letter fallback
AI recommendation quality degradesMediumMediumConfidence threshold + auto-fallback to rule-based playbooks
